\chapter{Algebraic Effect Handlers}
\label{chap:effects}

Algebraic Effects is the mathematical model for computational effects first introduced in \cite{plotkin2003algebraic} as an extension to the $\lambda$-calculus. After some time, Effect Handlers are introduced in \cite{plotkin2013handling} as a new addition to programming languages. Algebraic Effects and Handlers allows programmers to write programs with complex control flows. Functionalities such as exception handling, asynchronousity, I/O operations, and others are proven to be implement-able using Algebraic Effect Handlers. In this thesis, we use the term Algebraic Effect Handlers, Effect Handlers, and Handlers interchangably.

Algebraic Effect Handlers adapts the use of Call-By-Push-Value \cite{levy2013call} to separate value types, and computation types. This separation helps typing the computation together with possible effects, whereas value types are pure. This formal model usually provides reduction rules for its semantics.

In another spectrum, Effect Handlers are modeled by a combination of row types on top of \systemfw, named \systemfe \cite{xie2020effect}. The type rules are adjusted to accompany with a list of effects for an expression. This formal model usually provides evaluation contexts for its semantics.

\section{Language \lange}

In this section, we introduce \lange, a formal model of the Algebraic Effect Handlers as a model language for our target language for the compiler. Previously mentioned, we have two main choices for the formal language, (i) in the flavour of fine-grain call-by-value or (ii) in the flavour of \systemfe. We have chosen to adapt the (ii) flavour into the formal language due to its evaluation context semantics, which is helpful for our proofs.

We adopt and extend the language of \systemfe\ such that it can support all expressions available to \langc, including number type, binary operations, and if expression. We also introduce a new expression \eBlame{p} blaming the party \attrE{p}.

The language is polymorphic, with higher-kinded type system where effect is
treated as a different kind from basic type, where in traditional effect type
system, they are expressed as computational type. Effects are represented using
row types, each row is indexed by a label. The environment of effect labels
$\attrE{\Sigma}$ maps each labels to operations.

Programmers can define operations to be handled by rows of \attrE{h}, consisting of operation names, and their behaviors. A handler is also a value, and can be applied with a function ($\eHandler{h} \ \attrE{f}$) which a shorthand to a handle expression with $\attrE{f}$ applied to unit. In general, $\eHandle{h}{e}$ is used to run $\attrE{e}$ with effect handler $\attrE{h}$. During evaluation of $\attrE{e}$, an operation can be invoked by applying $\ePerform{op}{\sigma}$ to a value. If $\attrE{op}$ is declared in any handler of the current context, then the operation is handled using the closest handler. Because we define \lange\ well-typedness as no unhandled effects, there is no such case that an operation is undeclared once the program is well-typed.

A basic value, booleans and numbers, has no effect. A function $\attrE{\sigma_1\to\epsilon\sigma_2}$ receiving a type $\attrE{\sigma_1}$ producing a $\attrE{\sigma_2}$ value can invoke effects in $\attrE{\epsilon}$. A handler $\attrE{h} = \attrE{\{} \attrE{op_1 \to f_1}, \attrE{\dots}, \attrE{op_n \to f_n} \attrE{\}}$ must unify all operation types into one, allowing effect label of these operations, and other effects allowed by the environment. Because there is no type unification, therefore, these operations must return the same type.

Type judgement $\eTypesC{\eGamma}{e}{\sigma}{\epsilon}$ keeps a list of effects $\attrE{\epsilon}$ available to the expression $\attrE{e}$ being typed with type $\attrE{\sigma}$.

Following \cite{xie2020effect}, we adopt the \textit{dot notation} to decompose and compose evaluation contexts, as defined in Figure \ref{fig:lange-dot}. We would write $v \cdot \text{handle } h \cdot E \cdot e$ as a shorthand for $v \, (\text{handle } h \, (E[e]))$.

This thesis introduces two languages. To effectively distinguish the languages
used, we have chosen different typeset styling for both languages. \lange\ will
be typeset in \attrE{red\ serif} with  $\eStep$ is the reduction arrow, and
$\eGamma$ is the typing context.

\section{Soundness and Correctness}

Our algebraic effects handler system has type soundness by progress
(Theorem \ref{theorem:lange-progress}) and preservation (Theorem
\ref{theorem:lange-preservation}). Our effect system is also safe, i.e., all
effects are handled or specified in the judgement. To express safety of effects,
we first define (in Figure \ref{fig:lange-metafunc}), $\lceil \attrE{E}
\rceil^l$ extracts all labels in the context; and $\text{bop}(\attrE{E})$
extracts all $\attrE{op}$ in the context. Lemma \ref{lemma:lange-op-handled} and
Lemma \ref{lemma:lange-types-meaningful} express effect safety.


\begin{theorem}[\lange\ Progress]
  \label{theorem:lange-progress}

  If $\eTypesC{\cdot}{e_1}{\sigma}{\langle \rangle}$, then either $\attrE{e_1}$ is a value, or there exists an $\attrE{e_2}$ such that $\attrE{e_1} \eStep \attrE{e_2}$.
\end{theorem}

\begin{theorem}[\lange\ Preservation]
  \label{theorem:lange-preservation}

  If $\eTypesC{\cdot}{e_1}{\sigma}{\langle \rangle}$ and $\attrE{e_1} \eStep
  \attrE{e_2}$, then $\eTypesC{\cdot}{e_2}{\sigma}{\langle \rangle}$.
\end{theorem}

\begin{lemma}[Well-typed operations are handled]
  \label{lemma:lange-op-handled}

  If $\eTypesC{\cdot}{\attrE{E}[\ePerform{op}{\overline{\sigma}} \ \attrE{v}]}{\sigma}{\langle \rangle}$, then $\attrE{E}$ has the form $\attrE{E_1} \cdot \eHandler{h} \cdot \attrE{E_2}$ with $\attrE{op} \notin \text{bop}(\attrE{E_2})$ and $\attrE{op \to f} \in \attrE{h}$.
\end{lemma}

\begin{lemma}[Effects types are meaningful]
  \label{lemma:lange-types-meaningful}

  If $\eTypesC{\cdot}{\attrE{E}[\ePerform{op}{\overline{\sigma}} \ \attrE{v}]}{\sigma}{\epsilon}$ with $\attrE{op} \notin \text{bop}(\attrE{E})$, then $\attrE{op} \in \attrE{\Sigma}(l)$ and $l \in \attrE{\epsilon}$.
\end{lemma}
